# AIPM 后半段流转方案 v1

这份文档定义后半段核心流转：

1. 产品模型
2. PRD
3. 原型

目标是把前半段已经明确的需求结果，继续推进成正式文档和交互表达。

---

## 1. 为什么要单独定义后半段

前半段解决的是：

- 需求收集
- 需求整理
- 需求结构化
- 需求澄清

后半段解决的是：

- 怎么把这些结果写成正式文档
- 怎么把文档变成页面和流程表达

所以后半段的重点不再是“理解需求”，而是：

- 产品定义
- 正式文档
- 页面表达

---

## 2. 三阶段一句话定义

### 2.1 产品模型

把需求结果转成产品骨架。

### 2.2 PRD

把产品骨架写成正式产品文档。

### 2.3 原型

把产品骨架和 PRD 变成可查看的页面和交互表达。

---

## 3. 三阶段上下游关系

### 3.1 产品模型 -> PRD

`产品模型` 提供：

- 模块
- 页面
- 流程
- 页面职责

`PRD` 基于这些结果写成正式文档。

### 3.2 PRD -> 原型

`PRD` 提供：

- 功能需求
- 页面说明
- 用户流程
- 验收标准

`原型` 基于这些结果，把内容落成页面结构和交互表达。

---

## 4. 三阶段输入输出总表

### 4.1 产品模型

输入：

- 需求结构化文档
- 需求澄清文档

输出：

- 产品模型文档

### 4.2 PRD

输入：

- 产品模型文档
- 需求结构化文档
- 需求澄清文档

输出：

- PRD 文档

### 4.3 原型

输入：

- 产品模型文档
- PRD 文档

输出：

- 原型稿

---

## 5. 主 Agent 的统一规则

后半段仍然遵循统一主 Agent 原则：

### 5.1 先回答，再进入产物动作

### 5.2 产物更新必须显式触发

### 5.3 阶段推进必须显式确认

### 5.4 不输出内部调试结构

---

## 6. 三阶段推进条件

### 6.1 产品模型 -> PRD

条件：

- 模块、页面、流程已基本清楚
- 用户认可当前骨架可继续
- 用户确认进入 PRD

### 6.2 PRD -> 原型

条件：

- 已形成正式 PRD
- 页面和流程足够清楚
- 用户确认进入原型

### 6.3 原型 -> UI

条件：

- 页面结构稳定
- 关键流程稳定
- 用户确认当前原型可作为 UI 输入

---

## 7. 三阶段状态流转

每个阶段建议统一采用：

- 未生成
- 已生成
- 待更新
- 已确认

### 7.1 未生成

还没有正式产物。

### 7.2 已生成

已经有正式产物，但仍可继续修改。

### 7.3 待更新

上游结果变化，当前产物需要重新生成。

### 7.4 已确认

用户认可当前产物可作为下一阶段输入。

---

## 8. 用户动作流

### 8.1 产品模型阶段

用户会：

- 查看模块、页面、流程
- 问结构问题
- 调整模型
- 更新产品模型文档
- 确认进入 PRD

### 8.2 PRD 阶段

用户会：

- 阅读正式 PRD
- 提问或修改文档
- 更新 PRD
- 看版本
- 确认进入原型

### 8.3 原型阶段

用户会：

- 查看页面和流程
- 调整页面结构
- 更新原型
- 看版本
- 确认进入 UI

---

## 9. 系统动作流

### 9.1 产品模型

系统负责：

- 基于前序结果生成骨架
- 输出模块、页面、流程
- 保存模型文档

### 9.2 PRD

系统负责：

- 基于产品模型和前序文档生成正式 PRD
- 保持一致性
- 保存 PRD 文档

### 9.3 原型

系统负责：

- 基于产品模型和 PRD 生成原型稿
- 输出页面清单和流程
- 保存原型产物

---

## 10. 三阶段核心交付物

### 10.1 产品模型阶段

- 产品模型文档

### 10.2 PRD 阶段

- 正式 PRD 文档

### 10.3 原型阶段

- 原型稿

这三份结果共同构成后续 UI 的输入基础。

---

## 11. 第一版实现建议

第一版应围绕这三阶段做最小闭环：

1. 产品模型可生成正式骨架
2. PRD 可生成正式文档
3. 原型可生成页面和流程表达
4. 每阶段可编辑、版本化、显式推进

---

## 12. 结论

`产品模型 -> PRD -> 原型` 应该是 AIPM 后半段最核心的产品定义主链。

只要这三阶段跑顺，后面的 UI 才有稳定输入，而不会继续靠猜。
