# AIPM 阶段参考表 V1

## 1. 目的

本文档把 `AIPM` 的 9 个标准阶段整理成一张可直接用于产品设计、交互设计和技术实现的参考表。

重点回答 5 个问题：

- 每个阶段是做什么的
- 每个阶段的输入是什么
- 每个阶段的输出是什么
- 每个阶段谁来主导
- 每个阶段是否可以跳过

## 2. 阶段总览

| 阶段 | 主要目的 | 典型输入 | 主要输出 | 主导者 | 是否建议跳过 | 与上下游关系 |
| --- | --- | --- | --- | --- | --- | --- |
| 需求采集 | 收集原始需求和背景材料 | 一句话需求、文档、聊天记录、会议纪要 | `brief.md`、提取文本、来源记录 | 用户 + AI | 不建议 | 是整条链的起点 |
| 需求结构化 | 把原始输入整理成结构化需求点 | 原始需求、提取文本 | 结构化需求文档、范围、风险、待确认项 | AI 初稿，用户确认 | 不建议 | 为澄清和建模提供基础 |
| 需求澄清 | 识别缺口并补齐关键信息 | 结构化需求、缺失字段 | 澄清问题、回答、完整需求摘要 | AI 追问，用户回答 | 强烈不建议 | 决定后续 PRD 是否靠谱 |
| 产品模型 | 建立统一中间模型 | 完整需求结构、澄清结果 | 用户模型、功能模型、页面模型、流程模型 | AI 生成，用户修订 | 小项目可弱化，不建议省掉 | 是 PRD / 原型 / UI 的共同基底 |
| PRD | 形成正式产品需求文档 | 产品模型、完整需求 | `prd.md`、`prd.json`、验收标准、风险 | AI 起草，用户编辑 | 不建议 | 是原型和评审的文字基准 |
| 原型 | 把 PRD 转成页面和交互骨架 | PRD、页面模型、流程模型 | 页面原型、页面结构、流程骨架 | AI 生成，用户调整 | 不建议 | 承接 PRD，供 UI 和标注使用 |
| 原型标注 | 补充交互、业务、数据说明 | 原型、PRD、评审意见 | 标注文件、交互说明、异常状态说明 | 用户 + AI | 可在极简项目中合并到原型，但建议保留 | 让 UI 和研发理解更准确 |
| UI Draft | 生成图形化 UI 初稿 | 原型、原型标注、视觉约束 | UI 页面稿、视觉规范、组件建议 | AI 生成，用户调整 | 取决于是否需要设计交付 | 承接原型，进入审查与交付 |
| Review / 导出 | 审查、留痕、版本化和交付 | 各阶段产物、历史版本 | review 结果、diff、导出包、回滚点 | AI + 用户 | 不建议 | 是交付和闭环的最后一层 |

## 3. 各阶段详细说明

### 3.1 需求采集

**核心问题**

- 我们一开始到底收到了什么需求
- 需求来自谁、来源是什么、上下文是什么

**主要作用**

- 记录原始输入，不丢信息
- 支持上传文件并识别文本
- 给 AI 后续分析提供最初上下文

**典型输入**

- 一段文字
- doc / pdf
- 会议纪要
- 其他补充材料

**典型输出**

- `brief.md`
- `intake.json`
- `source-files/*`

**主导方式**

- 用户提供原始内容
- AI 做抽取、归纳和首轮理解

**是否可跳过**

- 不建议

**为什么不能轻易跳过**

- 如果原始输入没有被完整记录，后面所有阶段都可能建立在偏差理解上

### 3.2 需求结构化

**核心问题**

- 原始需求里真正的目标、场景、功能点是什么

**主要作用**

- 把口语化需求整理成结构化问题
- 先初步划清范围与非范围

**典型输入**

- 原始需求
- AI 提取内容

**典型输出**

- 用户目标
- 核心场景
- 核心功能点
- 范围 / 非范围
- 风险
- 待确认项

**主导方式**

- AI 先结构化
- 用户做删改和确认

**是否可跳过**

- 不建议

**为什么不能轻易跳过**

- 如果直接写 PRD，往往会把原始话术直接放大，缺少真正的产品拆解

### 3.3 需求澄清

**核心问题**

- 当前需求还缺哪些关键决策信息

**主要作用**

- 让 AI 主动追问
- 补齐关键字段和限制条件

**典型输入**

- 结构化需求
- 缺失字段
- 风险和待确认项

**典型输出**

- `clarify-question-pack.json`
- `clarify-answers.json`
- `clarify-summary.md`

**主导方式**

- AI 提问
- 用户回答
- 用户也可以手工编辑答案

**是否可跳过**

- 强烈不建议

**为什么不能轻易跳过**

- 很多需求不是错误，而是不完整；这一步省掉，后面产物质量会明显下降

### 3.4 产品模型

**核心问题**

- 有没有一个统一对象，把 PRD、原型、UI 串起来

**主要作用**

- 建立统一中间层
- 保证后续产物不各写各的

**典型输入**

- 完整需求结构
- 澄清答案

**典型输出**

- 目标用户模型
- 功能模型
- 页面模型
- 流程模型
- 设计约束

**主导方式**

- AI 起草模型
- 用户按业务理解做修订

**是否可跳过**

- 极小项目可以弱化，但不建议在系统设计里省掉

**为什么很重要**

- 这一步是上下游一致性的关键，没有它，PRD、原型、UI 很容易脱节

### 3.5 PRD

**核心问题**

- 怎么把产品定义正式沉淀成可对齐文档

**主要作用**

- 写出正式产品需求文档
- 提供评审、设计、研发的统一参考

**典型输入**

- 产品模型
- 澄清后的需求

**典型输出**

- `prd.md`
- `prd.json`
- 用户故事
- 验收标准
- 风险与开放问题

**主导方式**

- AI 初稿
- 用户持续编辑和 review

**是否可跳过**

- 不建议

**为什么不能轻易跳过**

- 如果没有 PRD，就缺少正式的产品定义载体，后续原型和 UI 很难有统一基准

### 3.6 原型

**核心问题**

- 产品的页面结构和交互骨架到底是什么样

**主要作用**

- 把文字需求变成页面骨架
- 明确页面关系和操作路径

**典型输入**

- PRD
- 页面模型
- 用户流程

**典型输出**

- 页面原型
- 页面结构
- 交互骨架

**主导方式**

- AI 先生成
- 用户继续改页面和布局

**是否可跳过**

- 不建议

**为什么不能轻易跳过**

- 没有这一步，UI 很容易直接跳成视觉稿，结构和流程容易出错

### 3.7 原型标注

**核心问题**

- 哪些业务规则、交互逻辑、异常状态无法只靠画面表达

**主要作用**

- 补充原型背后的说明
- 让设计、研发、测试理解更一致

**典型输入**

- 原型
- PRD
- review 意见

**典型输出**

- 标注文件
- 区块说明
- 交互说明
- 异常状态说明

**主导方式**

- 用户手工标注
- AI 生成标注建议

**是否可跳过**

- 极简项目可以合并到原型说明里
- 但对真实产品交付，建议保留

**为什么重要**

- 它是“从能看懂的图”到“能落地的设计说明”的关键一步

### 3.8 UI Draft

**核心问题**

- 最终页面视觉要长什么样

**主要作用**

- 在原型基础上生成图形化 UI 初稿
- 提供视觉方向和高保真表达

**典型输入**

- 原型
- 原型标注
- 风格约束
- 项目系统提示词

**典型输出**

- UI 页面稿
- 视觉风格说明
- 组件样式建议

**主导方式**

- AI 生成初稿
- 用户继续调整风格和细节

**是否可跳过**

- 如果当前目标只到产品定义，可以先不做
- 如果要形成设计交付，则不建议跳过

**为什么要谨慎定义**

- `UI Draft` 更适合定义成“高质量初稿”，不应承诺一次生成最终完稿

### 3.9 Review / 导出

**核心问题**

- 怎么确认质量、保留版本、做最终交付

**主要作用**

- 审查当前产物
- 记录版本和差异
- 支持导出和回滚

**典型输入**

- 各阶段产物
- 版本历史
- review 规则

**典型输出**

- review 结果
- diff
- 回滚点
- 导出包

**主导方式**

- AI review
- 用户确认和导出

**是否可跳过**

- 不建议

**为什么不能轻易跳过**

- 没有这一层，系统就只是“生成器”，不是“可交付的工作台”

## 4. 哪些阶段是最小闭环必须有

如果只做一个可落地最小闭环，建议保留：

- 需求采集
- 需求结构化
- 需求澄清
- PRD
- 原型
- Review / 导出

如果要保证上下游一致性，建议再加上：

- 产品模型

如果要支持更真实的设计交付，建议再加上：

- 原型标注
- UI Draft

## 5. 对 AIPM 的产品结论

对于 `AIPM` 这个产品，这 9 个阶段不一定意味着每个项目都必须严格逐个执行到死，但它们应该被定义成系统内的标准参考流程。

更准确地说：

- 它们是 `AIPM` 的默认标准路径
- 允许用户在部分项目里跳过某些阶段
- 但系统必须知道跳过的代价，并提醒影响

## 6. 交互设计建议

为了让这 9 个阶段在 UI 里可用，建议页面层做这些支撑：

- 每个阶段卡片上明确显示：
  - 当前状态
  - 主产物
  - 最近更新时间
  - 是否可进入
  - 是否可回退

- 每个阶段都支持 3 个基础动作：
  - 预览
  - 编辑
  - 继续通过 AI 补充

- 如果用户回退阶段，系统需要提示：
  - 哪些后续阶段会受影响
  - 是否标记为待重生成
  - 是否立即重跑后续阶段
