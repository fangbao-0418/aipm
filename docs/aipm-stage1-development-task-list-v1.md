# AIPM 第一阶段开发任务清单 V1

## 1. 清单目标

这份清单只服务当前第一阶段：

- 收集散落需求点
- 收集评论、补充说明、回复
- 收集上传文件
- AI 整理成需求点文档
- 支持编辑
- 支持版本历史、diff、回滚
- 主 Agent 全程跟进回复

这份清单的用途是：

- 明确当前已经有的能力
- 明确第一阶段还缺的能力
- 明确下一步开发顺序

## 3. 当前已具备能力

### 3.1 项目与本地工作区

- CLI 可启动本地 workspace
- 可创建项目
- 项目可本地落盘
- 模型设置已有项目级保存能力

### 3.2 第一阶段来源采集

- 聊天输入可进入第一阶段
- 来源记录支持新增、编辑、删除、回复
- 文件可上传并保留到本地项目空间
- `txt / md / json / csv / pdf / doc / docx / xls / xlsx` 已有解析链路

### 3.3 需求点文档

- 需求点文档已支持富文本编辑
- 支持手工保存
- 支持 AI 整理
- 支持历史版本
- 支持回滚

### 3.4 聊天和流式反馈

- 聊天已有流式接口
- 已能显示状态流
- 已支持停止回复
- 已接上部分模型增量输出

### 3.5 主 Agent

- 主 Agent 已有服务层
- 可根据当前项目状态给出阶段判断
- 可把判断结果写入项目空间日志

## 4. 当前还缺的关键能力

### 4.1 产品体验层缺口

1. 主 Agent 的跟进回复还不够“产品专家化”
2. 第一阶段“何时可进入下一阶段”的提示还不够明确
3. 聊天里的模型实时输出还偏技术调试感，不够产品化
4. 需求点文档与来源记录之间还没有强关联
5. 文件预览体验还不统一

### 4.2 功能层缺口

1. 历史版本 `diff` 能力还需要更完整
2. 来源记录变更后的“未整理提示”还不明显
3. 主 Agent 的推进确认还没有形成完整动作链
4. 第一阶段完成后的正式“进入下一阶段”动作还未闭环
5. 当前第一阶段的 review 结果还没有单独产物化

### 4.3 工程层缺口

1. 第一阶段缺少统一任务状态与完成判定对象
2. 主 Agent 回复模板还没有正式配置化
3. 第一阶段 AI 子任务还缺更明确的日志和可观测性
4. 前端还缺第一阶段专用的“未保存 / 未整理 / 可推进”状态提示

## 5. 开发优先级

建议按 `P0 -> P1 -> P2` 推进。

## 6. P0 任务

P0 的标准是：做完以后，第一阶段可以真正独立成立。

### P0-1 主 Agent 跟进回复产品化

目标：

- 让主 Agent 在第一阶段的聊天回复更像产品专家，而不是状态机输出

任务：

- 统一主 Agent 第一阶段回复模板
- 区分：
  - 已收录
  - 仅讨论未收录
  - 当前识别到的重点
  - 当前缺口
  - 是否建议整理
  - 是否建议推进
- 把主 Agent 回复结果作为结构化对象返回前端

涉及：

- 后端
- Prompt
- 前端消息展示

### P0-2 第一阶段“未整理”状态

目标：

- 来源记录或文件变化后，明确提示“当前文档不是最新整理结果”

任务：

- 给第一阶段加 `sourceDirty` 状态
- 来源记录新增/编辑/删除/回复后标记 dirty
- 上传文件后标记 dirty
- 保存文档不清除 dirty
- 只有 AI 整理完成后清除 dirty
- UI 显示“来源有变更，建议重新整理”

涉及：

- 后端状态
- bundle 返回
- 前端提示

### P0-3 文件预览统一化

目标：

- 让第一阶段来源文件预览更稳定、更一致

任务：

- 明确 `text / pdf / image / office` 预览策略
- `pdf` 走内嵌预览
- `text / md / json / csv` 走文本预览
- `doc / xls` 显示提取正文 + 原文件入口
- 统一预览弹层样式

涉及：

- 前端预览组件
- 后端文件读取接口

### P0-4 需求点文档版本 diff 完整化

目标：

- 版本历史不仅能回滚，还要能明确查看差异

任务：

- 支持“当前版本 vs 历史版本”
- 支持“任意两个历史版本”
- UI 上清晰标记新增、删除、修改

涉及：

- 后端 diff 计算
- 前端 diff 展示

### P0-5 第一阶段推进确认闭环

目标：

- 第一阶段达到条件后，主 Agent 在聊天中明确提示是否进入下一阶段

任务：

- 定义第一阶段推进 gate
- 主 Agent 生成 `canAdvance / blockers / reasons`
- 前端在聊天中展示推进确认消息
- 用户确认后执行正式推进动作

涉及：

- 主 Agent
- API
- 前端聊天区

## 7. P1 任务

P1 的标准是：让第一阶段更好用、更可信。

### P1-1 来源记录与文档关联高亮

目标：

- 点击某条来源记录，能看到它影响文档哪部分

任务：

- 记录来源片段与文档段落关联
- UI 支持点击来源高亮文档段落

### P1-2 第一阶段 review 产物

目标：

- 第一阶段也要有轻量 review 结果

任务：

- 输出：
  - 当前是否存在明显遗漏
  - 是否有冲突信息
  - 是否还缺关键背景
- 保存 review 结果文件

### P1-3 主 Agent 运行日志面板

目标：

- 让第一阶段 AI 行为可观察

任务：

- 展示：
  - 当前使用模型
  - 是否走 fallback
  - 本次读了多少来源
  - 当前运行到了哪一步

### P1-4 第一阶段导出

目标：

- 用户可以把当前需求点文档导出

任务：

- 导出 Markdown
- 导出 HTML
- 导出 PDF

## 8. P2 任务

P2 的标准是：为后续阶段接入做准备。

### P2-1 第一阶段完成态固化

目标：

- 第一阶段输出成为后续阶段的稳定输入

任务：

- 生成标准化需求点对象
- 为下一阶段输出固定文件格式

### P2-2 第二阶段启动入口

目标：

- 第一阶段结束后，能进入第二阶段

任务：

- 确认推进后初始化第二阶段状态
- 生成第二阶段初始任务

## 9. 推荐开发顺序

建议按这个顺序做，不要乱：

1. `P0-2` 第一阶段未整理状态
2. `P0-1` 主 Agent 跟进回复产品化
3. `P0-5` 第一阶段推进确认闭环
4. `P0-4` 版本 diff 完整化
5. `P0-3` 文件预览统一化
6. 再进入 `P1`

这个顺序的原因是：

- 先让第一阶段状态清楚
- 再让主 Agent 跟得住
- 再让推进动作成立
- 最后再优化体验

## 10. 建议直接开做的第一批任务

如果现在立刻开始补功能，我建议先直接做这 3 个：

1. 第一阶段 `sourceDirty` 状态和前端提示
2. 主 Agent 第一阶段回复结构化与产品化
3. 第一阶段推进确认闭环

这是最能直接提升产品可用性的三件事。

## 11. 完成标准

这一版清单执行到 P0 完成时，第一阶段应满足：

1. 用户能完整收集散落诉求、评论、文件
2. AI 能基于全部来源整理需求点文档
3. 来源变化后系统会明确提示文档未整理
4. 主 Agent 会持续跟进回复
5. 文档支持版本历史、diff、回滚
6. 第一阶段满足条件后，可由主 Agent 引导进入下一阶段

到这一步，第一阶段才算真正可用。
