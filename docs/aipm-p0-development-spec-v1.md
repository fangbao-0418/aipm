## 1. 规格目标

这份文档不再讨论“为什么做”，而是直接回答：

- 这 6 个点各自要实现什么
- 输入输出是什么

- UI 应该表现什么
- 什么样算做完

## 2. D1 聊天输入分类规则

### 2.1 目标

让主 Agent 能先判断用户当前是在：

- 提问
- 描述需求
- 下修改指令
- 触发文档生成动作

从而避免“每句话都被当成录入动作”。

### 2.2 四类输入定义

#### A. 提问

定义：

用户在询问事实、状态、解释、建议、区别、原因、下一步。

典型例子：

- 我现在有多少条需求？
- 为什么这个阶段要做需求结构化？
- 进入下一阶段会产出什么？
- 这个文档和 PRD 的区别是什么？

系统行为：

- 先正常回答问题
- 默认不修改正式文档
- 默认不推进阶段

#### B. 需求描述

定义：

用户在表达想法、补充诉求、增加背景、上传资料说明、描述目标。

典型例子：

- 我想做一个任务管理应用，帮助团队协作和追踪项目进度
- 这个产品主要面向小团队
- 需要支持文件共享和评论

系统行为：

- 先正常分析和回应
- 默认把它视作“来源信息候选”
- 不自动改正式文档
- 可以建议“是否整理进当前文档”

#### C. 修改指令

定义：

用户明确要求修改当前阶段正式产物，但不一定是整篇重生成。

典型例子：

- 把这一段合并进当前需求整理文档
- 帮我把这部分写得更清楚
- 把目标用户改成中小团队负责人

系统行为：

- 明确这是对当前正式产物的修改动作
- 可以要求确认
- 执行后更新当前阶段正式文档

#### D. 文档生成指令

定义：

用户明确要求生成、重整、更新某个阶段正式产物，或确认进入下一阶段。

典型例子：

- 整理成需求文档
- 生成需求结构化文档
- 重新整理当前文档
- 进入下一阶段

系统行为：

- 进入正式文档动作
- 生成或更新当前阶段产物
- 必要时提示确认

### 2.3 判定优先级

当一条输入同时像多种类型时，按这个顺序判定：

1. 文档生成指令
2. 修改指令
3. 提问
4. 需求描述

原因：

- 明确动作优先于语义猜测
- 修改优先于泛化回答
- 问题优先于普通需求描述

### 2.4 不应误判的场景

以下输入不能被误判成文档动作：

- 我现在有多少条需求？
- 为什么这个阶段这么慢？
- 原型阶段到底做什么？

以下输入不能只被当成普通回答：

- 整理成需求文档
- 把这段加入当前文档
- 更新需求结构化结果

### 2.5 UI 承接

用户不一定看到“输入分类结果”，但系统行为要体现出来：

- 提问时：聊天正常答复
- 需求描述时：聊天自然回应，并可建议整理
- 修改指令时：聊天提示将修改当前文档
- 文档生成指令时：聊天提示开始生成或更新正式产物

### 2.6 验收标准

- 不再把普通提问误当成录入动作
- 不再把明确文档动作只当成闲聊回复
- 4 类输入都有稳定行为

## 3. D2 主 Agent 回复规范

### 3.1 目标

让主 Agent 的聊天回复像一个正常、专业、自然的 AI 产品专家，而不是流程机。

### 3.2 回复原则

#### 原则一：先回答用户

如果用户在问问题，先回答问题。

不能先输出：

- 我会收录这条内容
- 我会重新整理文档
- 我会推进下一阶段

除非用户明确要求执行这些动作。

#### 原则二：自然语言优先

聊天框中的回复只能是自然语言。

不应输出：

- JSON
- 调试结构
- 系统决策对象
- 内部任务计划对象

#### 原则三：产品语气

回复应具备：

- 清楚
- 简洁
- 有判断
- 不机械

不应具备：

- 系统腔
- 命令行腔
- 过度模板化总结

#### 原则四：动作和解释分开

如果用户在问“为什么”，就解释。

如果用户在说“帮我改”，就执行或确认执行。

不要把解释和执行混成一句系统话术。

### 3.3 允许的回复类型

#### A. 直接回答

用于提问场景。

例：

- “现在项目里有 12 条来源记录，其中 4 条已经整理进当前文档。”

#### B. 分析性回复

用于需求描述场景。

例：

- “这段内容已经能说明产品方向和核心价值，但目标用户还不够明确。我可以先帮你整理进需求文档，或者先继续补充。”

#### C. 动作确认回复

用于修改或生成动作。

例：

- “我会基于当前来源重新整理需求文档，并保留你刚补充的目标用户信息。”

#### D. 推进建议回复

用于阶段足够稳定时。

例：

- “当前需求整理文档已经比较完整，如果你认可，我们可以进入需求结构化阶段。”

### 3.4 禁止的回复类型

- “这条我会收录进需求采集文档，并重新整理当前需求点。”
- “已接收 96 条散落需求点。”
- “模型整理摘要：……”
- “当前由主 Agent 跟进这一阶段……”

这类回复最大的问题是：

- 不像正常交流
- 抢在用户前面做动作
- 系统味过重

### 3.5 UI 承接

聊天区只显示正常回复消息。

不额外显示：

- 主 Agent 卡片
- 采集摘要卡片
- 当前识别结果卡片
- JSON 展开面板

### 3.6 验收标准

- 回复像正常模型，不像流程机器人
- 没有 JSON、调试信息、系统卡片
- 提问、解释、执行、建议四种回复风格清楚

## 4. D4 显式文档动作清单

### 4.1 目标

定义当前系统允许哪些“正式动作”，让用户知道什么时候会修改正式产物。

### 4.2 当前支持的动作

#### A. 生成需求整理文档

触发方式：

- 整理成需求文档
- 生成需求整理文档
- 根据这些内容生成文档

作用：

- 基于当前来源生成第一阶段正式产物

#### B. 更新当前需求整理文档

触发方式：

- 重新整理当前文档
- 根据刚才内容更新当前文档
- 把这部分并入当前文档

作用：

- 更新当前阶段正式产物

#### C. 生成需求结构化文档

触发方式：

- 生成需求结构化
- 进入需求结构化阶段
- 根据当前需求文档提炼结构化结果

作用：

- 基于第一阶段正式产物生成第二阶段正式产物

#### D. 更新当前阶段文档

触发方式：

- 改写当前文档的某一部分
- 合并指定内容进当前文档
- 用更专业的表达重写某段

作用：

- 修改当前阶段正式产物

#### E. 进入下一阶段

触发方式：

- 进入下一阶段
- 继续到需求结构化
- 继续到需求澄清

作用：

- 基于当前阶段稳定结果进入下一阶段

### 4.3 动作确认规则

默认规则：

- 生成新文档：可直接执行
- 更新当前文档：可直接执行
- 进入下一阶段：需要明确确认
- 覆盖性较强的整篇重整：建议二次确认

### 4.4 动作结果反馈

动作执行后，聊天里只需要反馈：

- 做了什么
- 结果在哪里看
- 下一步可以做什么

例如：

- “需求整理文档已经更新好了，你可以现在打开查看。”
- “需求结构化文档已经生成，如果你认可，我们可以继续进入需求澄清。”

### 4.5 不应自动触发的动作

以下场景不能自动生成或更新正式文档：

- 普通问答
- 用户表达感受
- 用户评论系统体验
- 用户讨论阶段意义

### 4.6 验收标准

- 文档动作清单固定且清楚
- 用户知道什么时候会改文档
- 系统不会因普通聊天误触发正式动作

## 5. D7 需求整理文档结构

### 5.1 目标

定义第一阶段正式产物“需求整理文档”的结构。

这份文档的重点不是固定分类，而是：

- 保真
- 可读
- 可继续讨论
- 可继续整理

### 5.2 文档定位

需求整理文档是：

- 第一阶段正式产物
- 从散乱来源整理出来的第一份正式文档

它不是：

- PRD
- 结构化表格
- 系统摘要卡片

### 5.3 建议文档结构

#### 1. 文档标题

建议形式：

- `<项目名> 需求整理文档`

#### 2. 文档说明

简短说明这份文档是什么：

- 基于当前聊天、记录和上传资料整理
- 以保留原始需求语义为主

#### 3. 核心需求内容

这是主体部分。

要求：

- 尽量按语义层级整理
- 尽量保留原始需求点
- 允许合并相似项
- 允许把评论整理成更清楚的表达

这一部分不要求固定成：

- 目标与价值
- 规则与约束
- 其他需求点

而应根据当前项目内容自然整理成合适的小节。

#### 4. 当前待进一步确认的点

允许单独列出当前还不够清楚的内容。

但这不是主体，主体仍然是完整需求内容。

### 5.4 文档形式要求

- 以富文本方式呈现
- 支持标题、段落、列表
- 支持后续编辑
- 支持版本化

### 5.5 UI 承接

打开第一阶段文档时：

- 主区域显示需求整理文档
- 右侧显示来源记录、上传文件、版本

### 5.6 验收标准

- 文档不是只剩摘要
- 文档主体是完整需求内容
- 文档结构自然，不是硬套固定桶

## 6. D8 需求整理规则

### 6.1 目标

定义 AI 在第一阶段整理文档时到底做什么、不做什么。

### 6.2 应做的事

#### A. 合并重复表达

如果多个来源在表达同一件事，可以合并成一条更清楚的内容。

#### B. 整理层级

把散乱平铺的信息整理成更好读的层级关系。

#### C. 清洗表达

可以把过于口语、碎片化的内容改成更清楚的表达。

#### D. 保留冲突信息

如果来源之间有冲突，不应擅自删掉，而应明确标记存在不同说法。

#### E. 标出待确认点

可以把当前还不明确的内容单独列出。

### 6.3 不应做的事

#### A. 不应过度摘要

不能把大量需求点压缩成几个抽象桶。

#### B. 不应提前写成 PRD

第一阶段不是正式 PRD 写作阶段。

#### C. 不应硬套固定模板

不能无论什么内容都套：

- 目标与价值
- 规则与约束
- 其他需求点

#### D. 不应篡改用户原意

润色可以，但不能改变核心意思。

### 6.4 来源处理规则

#### 聊天来源

- 只有在明确动作下，才被整理进正式文档

#### 上传文件

- 作为来源参与整理
- 不直接原样塞进聊天回复

#### 评论和补充说明

- 可以整理进正式文档
- 也可以保留为“当前讨论意见”

### 6.5 整理后的结果反馈

整理完成后，聊天里的反馈应简洁：

- 已更新需求整理文档
- 当前重点补充了什么
- 还缺什么可以继续补

而不是：

- 输出内部统计
- 输出系统摘要对象

### 6.6 验收标准

- 整理结果更清楚，但不失真
- 大部分有效需求点都被保留
- 用户能认出“这还是我原来的需求”

## 7. D11 需求结构化文档结构

### 7.1 目标

定义第二阶段正式产物“需求结构化文档”的内容结构和存在价值。

### 7.2 文档定位

需求结构化文档是：

- 第二阶段正式产物
- 从需求整理文档中提炼出来的结构化输入

它的目的不是重写一遍需求整理文档，而是给后续阶段提供稳定输入。

### 7.3 建议文档结构

#### 1. 产品目标

回答：

- 这个产品要解决什么问题
- 希望达成什么结果

#### 2. 目标用户

回答：

- 谁是主要用户
- 谁是次要用户

#### 3. 核心场景

回答：

- 用户在什么场景下使用这个产品

#### 4. 核心动作

回答：

- 用户会执行哪些关键动作

#### 5. 功能点

回答：

- 产品必须提供哪些功能

#### 6. 范围边界

回答：

- 当前做什么
- 当前不做什么

#### 7. 依赖与约束

回答：

- 资源、规则、外部条件、限制是什么

#### 8. 风险与待确认项

回答：

- 哪些内容还不稳定
- 哪些点需要进入需求澄清阶段继续解决

### 7.4 和需求整理文档的关系

需求整理文档偏“保真整理”。

需求结构化文档偏“设计输入”。

所以第二阶段要做的不是：

- 再写一遍长文

而是：

- 把第一阶段内容提炼成后续可复用结构

### 7.5 UI 承接

打开第二阶段文档时：

- 主区域显示结构化文档
- 右侧可显示来源依据、版本、待确认项

### 7.6 验收标准

- 用户能一眼看懂第二阶段的价值
- 第二阶段产物能直接为需求澄清、产品模型服务
- 第二阶段与第一阶段边界清晰

---

## 8. 当前建议的实现顺序

如果现在立刻进入实现，我建议严格按这个顺序做：

1. `D1`
2. `D2`
3. `D4`
4. `D7`
5. `D8`
6. `D11`

原因：

- 先把聊天行为拉正
- 再把正式动作定义清楚
- 再把第一阶段和第二阶段正式产物定义清楚

这条顺序最稳，也最不容易返工。
