# AIPM 前三阶段 UI / 交互流转稿 v1

这份文档只定义前三阶段的 UI 与交互流转：

1. 需求整理
2. 需求结构化
3. 需求澄清

目标是把这三阶段在界面上的承接方式讲清楚，避免后续 UI 再跑偏。

---

## 1. 总体交互模型

AIPM 当前阶段采用固定三栏：

- 左侧：项目区
- 中间：聊天区
- 右侧：阶段区

但真正的阶段工作不在右侧直接完成，而是在点击阶段产物后，进入全屏工作区。

所以交互主线是：

1. 在聊天里沟通
2. 在右侧看阶段状态
3. 打开全屏工作区处理当前阶段正式产物

---

## 2. 中间聊天区规则

聊天区在前三阶段承担统一职责：

- 正常问答
- 补充需求
- 请求整理 / 结构化 / 澄清
- 请求修改当前阶段文档
- 获取主 Agent 建议

聊天区不承担：

- 正式文档主内容展示
- 调试信息展示
- JSON 展示

聊天区的输出应全部是自然语言。

---

## 3. 右侧阶段区规则

右侧阶段区负责：

- 展示阶段顺序
- 展示阶段状态
- 展示阶段产物入口
- 提供回到某阶段继续修改的入口

右侧阶段区不负责：

- 展示复杂内容细节
- 替代主工作区

---

## 4. 第一阶段：需求整理 UI 流转

### 4.1 阶段目标

把散乱来源整理成正式的需求整理文档。

### 4.2 中间聊天区

用户可以：

- 输入需求
- 上传文本或文件
- 提问
- 要求整理文档

主 Agent 会：

- 正常回答
- 说明理解到的内容
- 在用户明确要求时整理文档

### 4.3 右侧阶段区

展示：

- 需求整理阶段状态
- 需求整理文档入口
- 是否待重新整理

### 4.4 全屏工作区

点击 `需求整理文档` 后打开全屏工作区。

布局：

- 左侧主区：需求整理文档
- 右侧侧栏：需求点记录、上传文件、版本

用户可做：

- 编辑文档
- 预览文档
- 保存文档
- AI 整理
- 查看版本
- 查看 diff
- 回滚版本
- 新增 / 编辑 / 删除 / 回复来源记录
- 预览上传文件

### 4.5 推进规则

当用户确认当前整理文档足够清楚后，点击进入下一阶段。

---

## 5. 第二阶段：需求结构化 UI 流转

### 5.1 阶段目标

把需求整理文档转成结构化输入。

### 5.2 中间聊天区

用户可以：

- 问结构相关问题
- 让主 Agent 解释当前结构结果
- 要求重新结构化
- 要求修改结构内容

### 5.3 右侧阶段区

展示：

- 需求结构化阶段状态
- 结构化文档入口
- 是否待重新生成

### 5.4 全屏工作区

点击 `需求结构化文档` 后打开全屏工作区。

布局建议：

- 左侧主区：结构化文档
- 右侧侧栏：关键结构项、待确认项、版本

左侧主区应包含：

- 产品目标
- 目标用户
- 核心场景
- 核心动作
- 功能点
- 范围边界
- 约束与风险

用户可做：

- 编辑结构化文档
- 保存
- AI 重新结构化
- 查看版本
- 回滚

### 5.5 推进规则

当结构化结果基本清楚后，用户确认进入 `需求澄清`。

---

## 6. 第三阶段：需求澄清 UI 流转

### 6.1 阶段目标

把当前模糊点和缺口显式化并收敛。

### 6.2 中间聊天区

用户可以：

- 回答澄清问题
- 问为什么要问这些问题
- 追问 blocker
- 请求更新澄清文档

主 Agent 会：

- 提出关键问题
- 接收答案
- 解释影响
- 提示是否可以进入下一阶段

### 6.3 右侧阶段区

展示：

- 需求澄清阶段状态
- 澄清文档入口
- blocker 提示

### 6.4 全屏工作区

点击 `需求澄清文档` 后打开全屏工作区。

布局建议：

- 左侧主区：澄清文档
- 右侧侧栏：关键问题、已回答、未回答、blocker、版本

左侧文档应包含：

- 当前澄清目标
- 关键问题
- 已确认项
- 未确认项
- 假设
- blocker

用户可做：

- 编辑澄清文档
- 保存
- 根据当前回答重新整理
- 查看版本
- 回滚

### 6.5 推进规则

当关键 blocker 已解除，或主 Agent 判断已足够进入下一阶段时，由用户确认推进。

---

## 7. 三阶段的统一交互规则

### 7.1 每阶段都有正式产物

前三阶段都必须有自己的正式文档，不只是聊天记录。

### 7.2 每阶段都能重新生成

但重新生成必须是明确动作，不是隐式自动发生。

### 7.3 每阶段都可编辑

用户可以手工修正 AI 结果。

### 7.4 每阶段都版本化

至少支持：

- 版本列表
- diff
- 回滚

### 7.5 每阶段都由主 Agent 跟进

但主 Agent 的跟进留在聊天里，不额外做系统卡片堆在中间区。

---

## 8. 三阶段的关键状态提示

前三阶段都应有一致的状态语义：

- 未生成
- 已生成
- 待更新
- 已确认

其中：

- `待更新` 很重要，表示上游或来源变了，但正式产物还没更新
- `已确认` 表示用户认可可以作为下一阶段输入

---

## 9. 阶段切换交互

### 9.1 正向推进

用户从当前阶段点击“进入下一阶段”后，进入后续阶段。

### 9.2 回到前一阶段

用户可以回到前一阶段继续修改。

如果前一阶段改动影响当前阶段，应标记当前阶段：

- 待更新

### 9.3 不自动串行推进

系统不能自动从第一阶段一路跑到第三阶段。

每个阶段都需要用户确认。

---

## 10. 第一版实现建议

前三阶段的 UI 第一版应优先实现：

1. 三栏主布局
2. 阶段产物全屏工作区
3. 每阶段正式文档
4. 可编辑 / 预览
5. 版本列表 / diff / 回滚
6. 聊天区正常问答
7. 显式推进阶段

第一版先不追求：

- 复杂看板
- 额外辅助卡片
- 过多状态组件

---

## 11. 结论

前三阶段的 UI 和交互，不应做成“后台系统”，而应做成：

- 聊天负责协作
- 阶段区负责导航
- 全屏工作区负责处理正式产物

这条交互主线一旦定住，后面的产品模型、PRD、原型都可以按同一模式延续下去。
