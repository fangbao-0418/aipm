# AIPM 产品需求文档 v3

## 1. 产品概述

### 1.1 产品名称

AIPM

### 1.2 一句话定义

AIPM 是一个 `Chat-First` 的 AI 产品定义工作台，帮助用户把散乱需求逐步收敛成正式产品文档与交互表达。

### 1.3 产品定位

AIPM 不是一句话直接吐最终稿的生成器，也不是一个传统项目管理后台。

它更像一个持续协作的 AI 产品专家：

- 先正常理解和回答问题
- 再在明确动作下生成阶段产物
- 再陪用户逐阶段收敛产品定义

### 1.4 当前主流程

AIPM 当前围绕 6 个核心阶段构建主链路：

1. 需求整理
2. 需求结构化
3. 需求澄清
4. 产品模型
5. PRD
6. 原型

---

## 2. 产品目标

### 2.1 核心目标

帮助用户从散乱输入稳定走到正式产品方案，而不是停留在聊天记录或一次性生成结果。

### 2.2 当前版本目标

第一版的重点不是“自动一路生成最终 UI 稿”，而是把产品定义主链做稳。

具体包括：

- 聊天行为自然可用
- 阶段边界清楚
- 每阶段有正式产物
- 正式产物可编辑、可版本化、可回滚
- 阶段推进有明确门槛和确认动作

### 2.3 成功标准

如果用户可以稳定完成下面这条链路，说明产品方向成立：

1. 输入散乱需求或上传资料
2. 生成需求整理文档
3. 得到结构化结果
4. 明确关键待澄清问题
5. 建立产品模型
6. 生成正式 PRD
7. 生成原型稿

---

## 3. 目标用户

当前优先服务这几类用户：

- 独立开发者
- 创业团队负责人
- 产品经理
- 需要快速梳理产品方案的人

这些用户的共同特点：

- 想法很多，但输入往往不完整
- 需求来源分散，包含聊天、评论、文档、附件
- 需要一个 AI 协作者持续推进，而不是一次性吐结果
- 需要本地保存、可修改、可追溯的正式产物

---

## 4. 典型问题

当前用户真正遇到的问题不是“不会写 PRD”，而是：

- 需求输入太散
- 来源太多
- 评论和正式需求混在一起
- 需求讨论和正式文档之间断层明显
- 后续页面、交互、PRD 容易各自理解

所以 AIPM 解决的是三层问题：

1. 把输入收清楚
2. 把产品定义清楚
3. 把定义表达清楚

---

## 5. 产品原则

### 5.1 聊天先正常回答

用户提问时，主 Agent 必须先正常回答问题。

不能把每句话都理解成录入、推进或生成动作。

### 5.2 文档生成必须显式触发

正式产物的生成和更新，只能由明确动作触发。

例如：

- 用户说“整理成需求文档”
- 用户点击“生成 / 更新文档”
- 用户点击“进入下一阶段”

### 5.3 来源和正式产物严格分离

聊天内容、评论记录、上传文件是来源。

阶段文档、产品模型、PRD、原型稿才是正式产物。

### 5.4 每阶段都要有正式结果

每个阶段结束时，都必须形成一份正式产物，而不是只留下聊天上下文。

### 5.5 每阶段都可持续修改

正式产物必须支持：

- 手工编辑
- AI 协助修改
- 版本记录
- diff
- 回滚

### 5.6 阶段推进必须显式确认

主 Agent 可以建议推进，但不能默默自动进入下一阶段。

---

## 6. 核心交互模型

### 6.1 Chat-First

聊天是主入口。

用户可以在聊天里：

- 提问
- 输入想法
- 上传文本或文件
- 请求生成或更新阶段产物
- 请求解释当前阶段结果

### 6.2 三栏工作台

- 左侧：项目与导航
- 中间：聊天主区域
- 右侧：阶段与产物入口

### 6.3 全屏产物工作区

点击某阶段产物后，进入全屏工作区。

工作区职责：

- 主区域展示正式产物
- 侧栏展示来源、版本、状态、辅助信息

---

## 7. 六阶段主流程

### 7.1 阶段一：需求整理

目标：

把散乱来源整理成一份正式、可读、可编辑的需求整理文档。

输入：

- 聊天里的需求描述
- 评论和补充说明
- 上传文件
- 手工记录的来源内容

输出：

- 需求整理文档

这一阶段的价值：

先把“乱的内容”变成“人能读、能继续讨论的内容”。

### 7.2 阶段二：需求结构化

目标：

把需求整理文档转成后续设计和文档都可复用的结构化输入。

输入：

- 需求整理文档

输出：

- 需求结构化文档

建议包含：

- 产品目标
- 目标用户
- 核心场景
- 核心动作
- 功能点
- 边界
- 风险

这一阶段的价值：

把“可读文档”变成“可继续设计的输入”。

### 7.3 阶段三：需求澄清

目标：

识别缺口、冲突和模糊点，形成澄清结果。

输入：

- 需求结构化文档
- 用户补充回答

输出：

- 需求澄清文档

建议包含：

- 已确认项
- 未确认项
- 假设
- blocker
- 澄清问题

这一阶段的价值：

把“还不够确定的地方”显式化，防止后面阶段建立在错误假设上。

### 7.4 阶段四：产品模型

目标：

把需求结果转成产品骨架。

输入：

- 需求结构化文档
- 需求澄清文档

输出：

- 产品模型文档

建议包含：

- 功能模块
- 页面列表
- 用户流程
- 页面职责
- 信息架构

这一阶段的价值：

回答“这个产品从产品结构上到底长什么样”。

### 7.5 阶段五：PRD

目标：

把前面阶段结果整理成正式产品需求文档。

输入：

- 产品模型文档
- 需求结构化文档
- 需求澄清文档

输出：

- PRD 文档

建议包含：

- 背景
- 目标
- 用户
- 场景
- 功能需求
- 页面说明
- 用户流程
- 验收标准
- 风险与开放问题

这一阶段的价值：

把产品定义沉淀成正式可评审、可交付的文档。

### 7.6 阶段六：原型

目标：

把产品模型和 PRD 变成页面与交互表达。

输入：

- 产品模型文档
- PRD 文档

输出：

- 原型稿

建议交付形式：

- 页面列表
- 页面结构稿
- 页面流转关系
- 交互说明
- 原型标注

这一阶段的价值：

把“做什么”进一步变成“页面怎么承接、用户怎么操作”。

---

## 8. 阶段产物规则

### 8.1 每阶段只有一个正式主产物

每个阶段都应该有一个清晰的主产物：

- 需求整理文档
- 需求结构化文档
- 需求澄清文档
- 产品模型文档
- PRD 文档
- 原型稿

### 8.2 阶段产物不是聊天摘要

正式产物必须是独立可读、可继续编辑、可版本化的文件。

### 8.3 上游变更会影响下游

如果前面阶段产物被修改，后续阶段应被标记为“待更新”。

系统可以提示影响，但不自动重跑。

---

## 9. 主 Agent 统一行为

主 Agent 贯穿全部阶段，但行为必须一致。

### 9.1 对话时

- 正常回答问题
- 正常解释阶段
- 正常分析输入内容
- 不输出系统腔
- 不输出 JSON 给用户

### 9.2 面对需求内容时

- 可以识别哪些内容适合进入正式产物
- 可以建议“是否整理成文档”
- 不应默认把所有话都写进文档

### 9.3 面对阶段推进时

- 可以说明当前阶段是否已经足够稳定
- 可以建议是否进入下一阶段
- 必须等待用户确认

---

## 10. 阶段推进规则

### 10.1 推进前提

进入下一阶段前，至少满足：

- 当前阶段已有正式产物
- 当前阶段产物相对稳定
- 用户确认推进

### 10.2 推进顺序

- 需求整理 -> 需求结构化
- 需求结构化 -> 需求澄清
- 需求澄清 -> 产品模型
- 产品模型 -> PRD
- PRD -> 原型

### 10.3 回退规则

用户可以回到上游阶段修改内容。

如果回退修改，上游变更会让下游阶段变成“待更新”状态。

---

## 11. MVP 范围

第一版优先做稳这 6 个阶段，不追求后续完整 UI 出图链。

MVP 应包含：

1. 正常的聊天体验
2. 六阶段正式产物生成
3. 六阶段正式产物编辑
4. 版本历史
5. diff
6. 回滚
7. 显式阶段推进
8. 本地保存

当前不作为 MVP 核心承诺的内容：

- 一句话直接生成最终 UI 稿
- 无人确认的自动全流程推进
- 复杂多 Agent 编排
- 完整设计工具替代

---

## 12. 关键风险

### 12.1 来源保真风险

如果来源整理不完整，后续阶段都会跑偏。

### 12.2 聊天行为风险

如果主 Agent 不能正常回答问题，用户会觉得系统像流程机。

### 12.3 阶段边界风险

如果每个阶段做什么不清楚，用户会不知道为什么要经历这些步骤。

### 12.4 产物一致性风险

如果上游变更不能传递到下游状态，整个产物链会失真。

---

## 13. 产品结论

AIPM 当前最准确的产品定义是：

`一个由主 Agent 持续协作推进的 Chat-First 产品定义工作台。`

它通过 6 个阶段，把散乱需求逐步收敛成正式产品文档和交互表达。

这份文档应作为当前产品设计、技术设计和实现拆解的总基准。
