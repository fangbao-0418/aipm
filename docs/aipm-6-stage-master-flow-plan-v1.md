# AIPM 六阶段主流程方案 v1

这份文档定义 AIPM 当前最核心的一条完整主流程：

1. 需求整理
2. 需求结构化
3. 需求澄清
4. 产品模型
5. PRD
6. 原型

目标是把这 6 个阶段串成一条清楚、稳定、可落地的产品主线。

---

## 1. 为什么要定义这 6 个阶段

AIPM 的目标不是“用户输入一句话，系统直接吐最终稿”，而是：

- 先理解需求
- 再整理需求
- 再明确缺口
- 再建立产品骨架
- 再写正式文档
- 最后形成页面与交互表达

所以这 6 个阶段本质上在解决三件事：

1. 把输入变清楚
2. 把产品定义完整
3. 把定义变成可交付表达

---

## 2. 六阶段一句话定义

### 2.1 需求整理

把散乱来源整理成一份可读、可编辑、可版本化的需求整理文档。

### 2.2 需求结构化

把需求整理文档转成结构化输入，明确目标、用户、场景、功能、边界和风险。

### 2.3 需求澄清

把结构里还不清楚的部分显式化，形成待确认项、假设和 blocker。

### 2.4 产品模型

把已经较清楚的需求结果转成模块、页面、流程组成的产品骨架。

### 2.5 PRD

把产品骨架写成正式产品需求文档。

### 2.6 原型

把产品骨架和 PRD 变成页面结构、交互流程和原型表达。

---

## 3. 六阶段整体关系

可以把这条主线理解成三层。

### 3.1 理解层

- 需求整理
- 需求结构化
- 需求澄清

这三步解决：

- 信息有没有收全
- 信息有没有整理清楚
- 关键缺口有没有被识别和收敛

### 3.2 定义层

- 产品模型
- PRD

这两步解决：

- 产品骨架是否成立
- 正式文档是否完整

### 3.3 表达层

- 原型

这一步解决：

- 页面和交互怎么被表达出来

---

## 4. 六阶段输入输出总表

### 4.1 需求整理

输入：

- 聊天补充
- 来源记录
- 上传文件

输出：

- 需求整理文档

### 4.2 需求结构化

输入：

- 需求整理文档

输出：

- 需求结构化文档

### 4.3 需求澄清

输入：

- 需求结构化文档
- 用户补充回答

输出：

- 需求澄清文档

### 4.4 产品模型

输入：

- 需求结构化文档
- 需求澄清文档

输出：

- 产品模型文档

### 4.5 PRD

输入：

- 产品模型文档
- 需求结构化文档
- 需求澄清文档

输出：

- PRD 文档

### 4.6 原型

输入：

- 产品模型文档
- PRD 文档

输出：

- 原型稿

## 5. 六阶段的阶段职责

### 5.1 需求整理负责什么

负责：

- 收住输入
- 保留来源
- 生成第一份正式阶段文档

不负责：

- 写 PRD
- 设计页面

### 5.2 需求结构化负责什么

负责：

- 从整理文档里抽取结构信息
- 形成后续可复用输入

不负责：

- 继续自由整理长文
- 提前做产品骨架

### 5.3 需求澄清负责什么

负责：

- 找出关键缺口
- 追问高影响问题
- 标记假设和 blocker

不负责：

- 重新写全部需求

### 5.4 产品模型负责什么

负责：

- 建立模块
- 建立页面
- 建立流程

不负责：

- 写正式 PRD 文案
- 画最终视觉稿

### 5.5 PRD 负责什么

负责：

- 形成正式文档
- 把前面阶段结果整合为统一产品说明

不负责：

- 代替原型

### 5.6 原型负责什么

负责：

- 把产品定义落到页面结构和交互表达

不负责：

- 代替最终 UI 设计稿

---

## 6. 主 Agent 的统一行为

这 6 个阶段都遵循同一套主 Agent 原则。

### 6.1 先回答问题

用户问问题时，主 Agent 先正常回答。

### 6.2 再决定是否进入产物动作

只有用户明确要求，或点击了对应动作，才生成或更新正式产物。

### 6.3 不默认自动录入

聊天消息默认不是正式产物。

### 6.4 不默认自动推进

系统不能默默把阶段往后推。

### 6.5 推进必须显式确认

主 Agent 只能提示：

- 当前是否已经可以继续

但真正推进必须由用户确认。

---

## 7. 六阶段推进条件

### 7.1 需求整理 -> 需求结构化

条件：

- 已形成正式需求整理文档
- 当前文档与来源一致，或用户接受当前版本
- 用户确认推进

### 7.2 需求结构化 -> 需求澄清

条件：

- 已形成结构化文档
- 当前仍存在关键缺口，或用户主动要求进入澄清
- 用户确认推进

### 7.3 需求澄清 -> 产品模型

条件：

- blocker 已收敛
- 核心问题已有答案或可带假设继续
- 用户确认推进

### 7.4 产品模型 -> PRD

条件：

- 模块、页面、流程已基本清楚
- 用户认可当前骨架可进入正式文档
- 用户确认推进

### 7.5 PRD -> 原型

条件：

- PRD 已形成稳定文档
- 页面和流程足够清楚
- 用户确认推进

---

## 8. 六阶段统一状态

建议所有阶段采用一致的 4 状态：

- 未生成
- 已生成
- 待更新
- 已确认

### 8.1 未生成

还没有正式产物。

### 8.2 已生成

已有正式产物，但仍可继续修改。

### 8.3 待更新

上游变化，当前产物需要重新生成或整理。

### 8.4 已确认

用户认可当前产物可作为下一阶段输入。

---

## 9. 六阶段用户动作流

### 9.1 在需求整理阶段

用户会：

- 输入需求
- 上传文件
- 补充记录
- 点击 AI 整理
- 编辑文档
- 看版本
- 回滚
- 确认推进

### 9.2 在需求结构化阶段

用户会：

- 查看结构结果
- 问结构问题
- 重新结构化
- 编辑结构文档
- 看版本
- 确认推进

### 9.3 在需求澄清阶段

用户会：

- 查看关键问题
- 回答问题
- 调整假设
- 更新澄清文档
- 看 blocker
- 确认推进

### 9.4 在产品模型阶段

用户会：

- 查看模块、页面、流程
- 提问和调整模型
- 更新产品模型
- 确认推进

### 9.5 在 PRD 阶段

用户会：

- 阅读 PRD
- 提问
- 修改 PRD
- 更新 PRD
- 看版本
- 确认推进

### 9.6 在原型阶段

用户会：

- 查看页面和流程
- 调整页面结构
- 更新原型
- 看版本
- 确认是否进入后续 UI

---

## 10. 六阶段系统动作流

### 10.1 需求整理

系统负责：

- 保存来源
- 标记文档是否待整理
- 显式触发后生成正式文档

### 10.2 需求结构化

系统负责：

- 提取结构信息
- 生成结构化文档
- 标记待确认项

### 10.3 需求澄清

系统负责：

- 识别关键缺口
- 形成问题列表
- 更新已确认 / 未确认 / 假设 / blocker

### 10.4 产品模型

系统负责：

- 生成模块、页面、流程骨架
- 保存产品模型文档

### 10.5 PRD

系统负责：

- 基于前面阶段结果生成正式文档
- 保存 PRD

### 10.6 原型

系统负责：

- 基于产品模型和 PRD 生成原型稿
- 保存原型产物

- 需求结构化文档

这 6 份产物共同构成 AIPM 当前主流程的正式结果链。

---

## 12. 第一版实现建议

第一版真正值得做的是这条完整主链的最小可用闭环：

1. 聊天正常回答
2. 每阶段有正式产物
3. 每阶段产物可编辑
4. 每阶段产物可版本化
5. 每阶段可显式更新
6. 每阶段可显式推进

不要第一版就追求：

- 全自动从第一阶段跑到第六阶段
- 高度复杂的多 Agent
- 过多隐藏中间层

---

## 13. UI 承接原则

这 6 个阶段建议都使用同一套 UI 承接原则：

- 聊天区负责协作
- 阶段区负责导航
- 全屏工作区负责处理正式产物

这样交互会稳定，用户不会每到一个阶段都重新学习。

---

## 14. 结论

`需求整理 -> 需求结构化 -> 需求澄清 -> 产品模型 -> PRD -> 原型`

应该被视为 AIPM 当前最核心的一条完整产品主流程。

这条主流程的价值在于：

- 它不是一键生成器
- 它是一条逐步收敛、逐步定义、逐步表达的产品链路

只要这 6 个阶段跑顺，后面的 UI、标注、设计交付才会有稳定基础。
