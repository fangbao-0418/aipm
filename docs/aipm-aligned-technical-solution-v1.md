# AIPM 对齐版技术方案 v1

这份文档只回答 6 件事：

1. 聊天行为原则
2. 阶段定义
3. 每阶段输入输出
4. 产物形式
5. 什么时候生成文档
6. 什么时候推进阶段

---

## 1. 聊天行为原则

### 1.1 聊天是主入口

AIPM 的核心交互是聊天，不是流程按钮。

用户可以在聊天里做 4 类事情：

- 提问
- 输入需求
- 上传文本或文件
- 请求生成或修改某个阶段文档

### 1.2 主 Agent 先正常回答，再决定是否整理

主 Agent 必须像正常模型一样回答问题。

例如：

- 用户问“我现在有多少条需求”
- 主 Agent 应直接回答这个问题

而不是默认回复：

- “这条我会收录进需求采集文档”

也就是说，系统不能把每句话都先当成“录入动作”。

### 1.3 不默认自动录入

聊天内容不应默认进入正式产物。

聊天内容先属于“来源信息”。

只有在以下情况之一出现时，才进入整理链路：

- 用户明确说“整理成文档”
- 用户点击“整理文档”
- 用户明确说“把这段收录进去”
- 用户明确说“根据这些内容更新当前文档”

### 1.4 聊天输出必须是自然语言

主 Agent 在聊天框里的回复必须是自然语言，不输出调试信息，不输出 JSON，不输出内部状态结构。

聊天框里不应展示：

- 原始模型结构化返回
- 内部任务计划 JSON
- 系统决策 JSON
- 调试型技术字段

### 1.5 来源信息和正式产物分离

聊天消息、上传文件、评论、补充说明，都是来源。

正式文档是基于来源整理出来的产物。

两者不能混在一起。

---

## 2. 阶段定义

当前建议使用 6 个主阶段。

### 2.1 需求整理

目标：
把散落输入整理成一份可读、可编辑、可继续讨论的需求整理文档。

这一阶段强调“保真整理”，不强调固定模板分类。

### 2.2 需求结构化

目标：
把需求整理文档转成适合后续产品设计的结构化信息。

这一阶段强调“抽取结构”，不是重新写一遍需求。

### 2.3 需求澄清

目标：
识别缺失信息、矛盾信息、模糊点，形成待确认项和假设。

这一阶段强调“补洞”。

### 2.4 产品模型

目标：
把结构化需求转成产品骨架。

这一阶段强调：

- 页面
- 功能模块
- 用户流程
- 信息架构

### 2.5 PRD

目标：
形成正式产品需求文档。

这一阶段强调“文档交付”。

### 2.6 原型

目标：
把 PRD 和产品模型转成页面和交互表达。

这一阶段强调“页面结构和交互说明”，不是最终视觉。

---

## 3. 每阶段输入输出

### 3.1 需求整理

输入：

- 聊天中的需求描述
- 上传的文本和文件
- 手工补充的来源记录
- 评论、备注、会议纪要

输出：

- 一份需求整理文档
- 来源记录列表
- 上传文件列表
- 版本记录

### 3.2 需求结构化

输入：

- 需求整理文档
- 来源记录
- 上传文件

输出：

- 产品目标
- 目标用户
- 核心场景
- 核心动作
- 核心功能点
- 范围边界
- 依赖与约束
- 风险与待确认项

### 3.3 需求澄清

输入：

- 需求结构化结果
- 来源记录
- 用户补充回答

输出：

- 澄清问题列表
- 已确认项
- 未确认项
- 当前假设
- 阻塞项

### 3.4 产品模型

输入：

- 需求结构化结果
- 需求澄清结果

输出：

- 功能模块
- 页面列表
- 页面职责
- 用户流程
- 信息架构
- 核心对象关系

### 3.5 PRD

输入：

- 产品模型
- 需求结构化结果
- 需求澄清结果

输出：

- 正式 PRD 文档

建议包含：

- 背景
- 目标
- 用户
- 场景
- 功能需求
- 页面说明
- 用户流程
- 验收标准
- 风险与开放问题

### 3.6 原型

输入：

- 产品模型
- PRD

输出：

- 页面级原型
- 页面流转
- 交互说明
- 原型标注

---

## 4. 产物形式

### 4.1 需求整理

产物形式：

- 富文本文档
- 本地保存
- 可编辑
- 可版本化
- 可回滚

### 4.2 需求结构化

产物形式：

- 结构化文档
- 同时保留“可读文本视图”和“结构化字段视图”

### 4.3 需求澄清

产物形式：

- 澄清文档
- 问答列表
- 假设与 blocker 列表

### 4.4 产品模型

产物形式：

- 结构化产品骨架文档
- 模块图 / 页面树 / 流程图的文字化表达

### 4.5 PRD

产物形式：

- 正式文档
- 支持编辑、版本、导出

### 4.6 原型

产物形式：

- 页面列表
- 页面低保真稿
- 页面标注
- 页面流转关系

第一版建议原型交付形式：

- HTML 原型页
- 页面区块说明
- 标注侧栏

## 5. 什么时候生成文档

文档不应随每次聊天自动生成。

建议只在以下情况生成或更新正式文档：

### 5.1 需求整理文档

触发条件：

- 用户明确要求“整理文档”
- 用户点击“AI 整理”
- 用户要求“根据这些来源更新文档”

### 5.2 需求结构化文档

触发条件：

- 需求整理文档已经稳定
- 用户确认进入下一阶段

### 5.3 需求澄清文档

触发条件：

- 结构化结果存在明显缺失
- 用户要求“生成待确认问题”

### 5.4 产品模型

触发条件：

- 需求结构化完成
- 关键澄清项已处理到可继续设计

### 5.5 PRD

触发条件：

- 产品模型已确认
- 需求澄清已达到可写 PRD 的程度

### 5.6 原型

触发条件：

- PRD 已稳定到可以表达页面与交互

## 6. 什么时候推进阶段

推进阶段必须是明确动作，不能默默自动推进。

### 6.1 推进原则

只有当前阶段的正式产物已经存在，并且用户确认后，才进入下一阶段。

### 6.2 各阶段推进条件

#### 需求整理 -> 需求结构化

推进条件：

- 已有一份需求整理文档
- 用户确认“这份整理文档可以作为下一阶段输入”

#### 需求结构化 -> 需求澄清

推进条件：

- 已完成结构化抽取
- 系统识别到仍有关键缺失，或用户主动要求进入澄清

#### 需求澄清 -> 产品模型

推进条件：

- 关键缺失项已处理
- 当前信息足以搭建产品骨架

#### 产品模型 -> PRD

推进条件：

- 页面、模块、流程已清楚
- 用户确认开始生成正式 PRD

#### PRD -> 原型

推进条件：

- PRD 内容已经足以表达页面结构和交互

### 6.3 主 Agent 的职责

主 Agent 的职责不是自动把阶段推进掉。

主 Agent 应该做的是：

- 告诉用户当前阶段是否已经可以继续
- 说明为什么可以继续，或者为什么还不能继续
- 给出建议动作
- 等用户确认后再推进

---

## 结论

新的产品方向应当是：

- 聊天先回归正常模型行为
- 文档整理作为明确动作触发
- 阶段用于产出不同层级的结果
- 阶段推进必须显式确认
- 每个阶段都要有明确产物，而不是系统内部概念

当前最优先做对的是：

1. 需求整理
2. 需求结构化
3. 需求澄清

先把这 3 步跑顺，再继续往 PRD 和原型推进。
