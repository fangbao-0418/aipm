# AIPM PRD 阶段产品方案 v1

这份文档只定义 `PRD` 这一阶段。

目标是把前面已经明确过的需求、结构、澄清结果和产品模型，整理成一份正式的产品需求文档。

---

## 1. 阶段定位

`PRD` 是 AIPM 在 `产品模型` 之后的正式文档阶段。

这一阶段不再解决“需求是什么”，也不再主要解决“产品结构是什么”，而是负责把前面已经相对稳定的内容写成一份正式、清晰、可评审、可交付的文档。

一句话定义：

`把已经清楚的产品骨架，写成一份正式产品需求文档。`

---

## 2. 这一阶段要解决的问题

即使已经有了：

- 需求整理
- 需求结构化
- 需求澄清
- 产品模型

仍然会存在这些问题：

- 信息散在不同阶段文档里，不适合直接评审
- 页面、流程、功能和验收标准还没汇总成一份正式文档
- 设计、研发、测试拿不到统一的产品定义

所以 PRD 阶段解决的是：

1. 把前序结果整合成统一正式文档
2. 把产品定义写清楚
3. 为原型和后续交付提供稳定依据

---

## 3. 阶段目标

### 3.1 用户目标

- 我希望拿到一份真正能看、能审、能交付的 PRD
- 我希望 PRD 不是凭空生成，而是基于前面阶段结果
- 我希望还能继续改，不是一次性定死

### 3.2 系统目标

- 为原型阶段提供明确输入
- 为设计、研发、评审提供正式文档
- 保持和前序阶段结果一致

---

## 4. 这一阶段的输入

主输入包括：

- `需求结构化文档`
- `需求澄清文档`
- `产品模型文档`

可参考：

- `需求整理文档`

原则上，PRD 不应脱离前序阶段单独生成。

---

## 5. 这一阶段的正式产物

这一阶段建议只有一个核心正式产物：

- `PRD 文档`

这是一份正式文档，不是摘要卡片，也不是系统自动报告。

---

## 6. PRD 文档应该输出什么

第一版建议至少包含 9 类内容。

### 6.1 背景

说明：

- 为什么做这个产品
- 当前需求的背景是什么

### 6.2 产品目标

说明：

- 当前产品要解决什么问题
- 当前阶段要达到什么目标

### 6.3 目标用户

说明：

- 主要用户是谁
- 次要用户是谁

### 6.4 核心场景

说明：

- 主要使用场景
- 关键使用流程

### 6.5 功能需求

说明：

- 当前明确要做的功能
- 功能优先级

### 6.6 页面与结构说明

说明：

- 主要页面
- 页面职责
- 页面与功能关系

### 6.7 用户流程

说明：

- 用户关键流程
- 流程起点和终点

### 6.8 验收标准

说明：

- 什么算阶段目标达成
- 功能到什么程度算完成

### 6.9 风险与开放问题

说明：

- 当前风险
- 尚未解决的问题

---

## 7. PRD 原则

### 7.1 PRD 是正式交付文档

PRD 的语气和结构都应更正式，不应像聊天整理稿。

### 7.2 PRD 不应脱离来源自由发挥

PRD 可以整合和润色，但不应脱离前序阶段凭空扩写。

### 7.3 PRD 重点在清楚和可执行

这一阶段不追求“写得像咨询报告”，而追求：

- 清楚
- 可评审
- 可继续设计

### 7.4 保留风险和开放问题

PRD 不是所有内容都必须完全确定。

如果仍有不确定项，应该明确写出来。

---

## 8. 主 Agent 在这一阶段怎么工作

### 8.1 主 Agent 的职责

主 Agent 在 `PRD` 阶段主要做 4 件事：

1. 解释 PRD 内容
2. 回答用户对文档的疑问
3. 根据用户要求修改 PRD
4. 判断是否可以进入原型阶段

### 8.2 聊天行为

例如用户问：

- “这个 PRD 里范围是不是太大了”
- “验收标准写得够不够”
- “这个场景是不是还没说清”

主 Agent 应直接回答。

如果用户说：

- “生成 PRD”
- “根据当前内容更新 PRD”
- “把 PRD 改得更正式一点”

主 Agent 才进入 PRD 更新动作。

### 8.3 不默认自动重写 PRD

普通讨论不应自动改写正式 PRD。

只有用户显式要求，或点击相关动作时，才更新正式产物。

---

## 9. 什么时候生成 PRD

### 9.1 允许触发

- 用户确认从产品模型进入 PRD
- 用户点击“生成 PRD”
- 用户在聊天里要求“生成 / 更新 PRD”

### 9.2 不应自动触发

- 普通问答
- 单条补充说明
- 只是在阅读前序文档时

---

## 10. 什么时候更新 PRD

更新 PRD 应当是显式动作。

### 10.1 当上游变化后

如果：

- 需求结构化变化
- 澄清结果变化
- 产品模型变化

则 PRD 应被标记为：

- 待更新

但不自动重写。

### 10.2 当用户要求更新后

只有用户明确要求，才重新生成正式 PRD。

---

## 11. UI 交互方案

### 11.1 主工作区

建议延续前面阶段的交互模型：

- 左侧主区：PRD 文档
- 右侧辅助区：章节结构、风险、开放问题、版本

### 11.2 左侧主区

要求：

- 正式文档视图
- 可编辑
- 可保存
- 可重新生成

### 11.3 右侧辅助区

建议承载：

- 章节目录
- 当前风险
- 当前开放问题
- 版本记录

### 11.4 聊天区

聊天区负责：

- 回答 PRD 问题
- 接收修改指令
- 解释 PRD 改动
- 提示是否可以进入原型阶段

---

## 12. 与下一阶段的关系

`PRD` 的主要去向是：

- `原型`

进入原型阶段的前提是：

- PRD 已形成稳定文档
- 页面和流程已经足够清晰
- 用户认可 PRD 可以作为原型输入

---

## 13. 完成标准

当满足以下条件时，可认为 `PRD` 阶段完成：

1. 已形成正式 PRD 文档
2. 背景、目标、用户、场景、功能、页面、流程、验收标准、风险已基本清楚
3. 当前 PRD 与上游阶段保持一致
4. 用户认可可进入原型阶段

---

## 14. 第一版范围建议

第一版优先做：

- 正式 PRD 文档
- 可编辑
- 可重新生成
- 版本记录
- 风险与开放问题

第一版先不追求：

- 很复杂的自动评审体系
- 自动竞品分析
- 复杂模板市场

---

## 15. 结论

`PRD` 阶段的意义，是把前面阶段沉淀的内容整合成一份正式交付文档。

只要这一步稳定，原型阶段就不再是“看着聊天和零散文档瞎画”。
