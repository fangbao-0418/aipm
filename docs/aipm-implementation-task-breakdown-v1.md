## 1. 拆解原则

### 1.1 先做主链，不先做边角

当前实现优先级必须围绕这条主链展开：

1. 正常聊天
2. 需求整理文档
3. 需求结构化文档
4. 需求澄清文档
5. 产品模型文档
6. PRD 文档
7. 原型稿

### 1.2 先做“可用”，再做“自动化”

第一版优先目标：

- 用户可以稳定走完阶段
- 每阶段能产出正式结果
- 每阶段都能编辑、保存、版本化

不是优先做：

- 全自动串行跑完
- 复杂多 Agent 编排
- 高级 UI 设计能力

### 1.3 主 Agent 是唯一对外 Agent

当前产品层面只保留一个主 Agent 概念。

后端内部可以有流程服务，但前端和产品表达不再暴露“多个 Agent”。

## 2. 总体任务分组

当前实现任务建议拆成 8 个任务组：

1. 基础架构与项目空间
2. 聊天主链
3. 需求整理阶段
4. 需求结构化阶段
5. 需求澄清阶段
6. 产品模型阶段
7. PRD 与原型阶段
8. 通用能力

---

## 3. 任务组一：基础架构与项目空间

### 3.1 目标

为所有阶段提供稳定的本地项目空间、产物存储、版本管理基础能力。

### 3.2 任务

#### T1. 项目空间目录统一

内容：

- 统一项目目录结构
- 统一阶段产物目录结构
- 统一来源文件目录结构
- 统一版本目录结构

建议目录：

- `workspace/projects/<projectId>/project.json`
- `workspace/projects/<projectId>/sources/`
- `workspace/projects/<projectId>/stages/requirement-organization/`
- `workspace/projects/<projectId>/stages/requirement-structuring/`
- `workspace/projects/<projectId>/stages/requirement-clarification/`
- `workspace/projects/<projectId>/stages/product-model/`
- `workspace/projects/<projectId>/stages/prd/`
- `workspace/projects/<projectId>/stages/prototype/`

验收标准：

- 新建项目后目录结构稳定
- 每阶段产物有明确落盘位置

#### T2. 项目元数据模型统一

内容：

- 定义项目基础信息
- 定义当前阶段
- 定义阶段状态
- 定义“待更新”状态
- 定义模型设置

验收标准：

- 项目状态可以稳定读写
- 切换阶段不会丢状态

#### T3. 通用版本能力

内容：

- 每次正式保存生成版本
- 支持查看历史版本
- 支持 diff
- 支持回滚

验收标准：

- 任一阶段产物都能看到历史版本
- 回滚后内容正确恢复

优先级：

- `P0`

## 4. 任务组二：聊天主链

### 4.1 目标

让聊天真正像正常模型对话，同时能在明确动作下驱动阶段产物生成和更新。

### 4.2 任务

#### T4. 聊天输入分类

内容：

把输入分成 4 类：

- 提问
- 需求描述
- 修改指令
- 文档生成指令

验收标准：

- 用户问问题时，主 Agent 正常回答
- 不再默认把每句话都当录入动作

#### T5. 主 Agent 自然语言输出规范

内容：

- 禁止在聊天里输出 JSON
- 禁止输出系统卡片堆叠
- 禁止输出调试字段
- 回复必须是自然语言

验收标准：

- 聊天框中只出现自然语言回复
- 不再出现“已收录 96 条需求点”这类系统腔主回复

#### T6. 聊天显式文档动作

内容：

- 只有用户明确触发时，才整理或更新正式文档
- 聊天支持“整理成需求文档”“更新当前文档”“进入下一阶段”这类明确动作

验收标准：

- 普通提问不会修改正式文档
- 明确动作才触发文档更新

#### T7. 聊天流式输出

内容：

- 聊天回复实时输出
- 上传文件处理过程可见
- 失败原因可见

验收标准：

- 用户输入后立即有反馈
- 不再出现长时间无反馈只转圈

优先级：

- `P0`

依赖：

- `T1`
- `T2`

---

## 5. 任务组三：需求整理阶段

### 5.1 目标

把散乱来源整理成一份正式、保真、可编辑的需求整理文档。

### 5.2 任务

#### T8. 来源收集模型统一

内容：

- 聊天来源
- 手工记录来源
- 上传文件来源
- 评论与补充来源

全部作为来源信息统一保存。

验收标准：

- 所有来源都能在项目空间中找到
- 来源和正式文档严格分离

#### T9. 文件上传与预览

内容：

- 支持文本文件上传
- 支持 PDF 预览
- 支持 DOC / DOCX 内容预览方案
- 上传文件列表只显示文件信息，不直接摊解析文本

验收标准：

- 文件上传后可在来源侧栏看到
- 点击文件可预览

#### T10. 需求整理文档生成

内容：

- 基于来源整理出正式需求整理文档
- 保留原始信息语义
- 以整理、去重、层级优化为主
- 不强行套固定模板分类

验收标准：

- 文档内容不再只剩摘要桶
- 来源中的关键信息能尽量保真进入文档

#### T11. 需求整理文档编辑与版本

内容：

- 文档富文本编辑
- 手工保存
- AI 协助修改
- 版本历史
- diff
- 回滚

验收标准：

- 文档可直接编辑
- 每次保存生成可回看的版本

#### T12. 进入需求结构化的推进门

内容：

- 当需求整理文档存在且相对稳定时，允许进入下一阶段
- 上游来源变更后，如未重新整理，提示用户先更新文档

验收标准：

- 推进规则清楚
- 用户知道为什么当前能或不能进入下一阶段

优先级：

- `P0`

依赖：

- `T4`
- `T6`
- `T7`

---

## 6. 任务组四：需求结构化阶段

### 6.1 目标

把需求整理文档转成后续阶段可以直接复用的结构化结果。

### 6.2 任务

#### T13. 需求结构化文档定义

内容：

明确定义结构化文档包含哪些内容：

- 产品目标
- 目标用户
- 核心场景
- 核心动作
- 功能点
- 边界
- 依赖与约束
- 风险与待确认项

验收标准：

- 结构化文档具备稳定字段和可读表达

#### T14. 从需求整理文档生成结构化结果

内容：

- 支持显式生成结构化文档
- 支持对结构化结果再次整理

验收标准：

- 用户能看懂“为什么从整理文档进入结构化文档”
- 结果可被下游直接使用

#### T15. 结构化文档编辑与版本

内容：

- 结构化文档支持手工编辑
- 支持 AI 协助调整
- 支持版本 / diff / 回滚

验收标准：

- 结构化结果不是一次性只读输出

#### T16. 进入需求澄清的推进门

内容：

- 用户确认结构化结果后，进入需求澄清

验收标准：

- 推进动作显式
- 结构化不完整时能提示继续补充

优先级：

- `P0`

依赖：

- `T10`
- `T11`

---

## 7. 任务组五：需求澄清阶段

### 7.1 目标

找出缺失点、冲突点、模糊点，并形成可继续推进的澄清结果。

### 7.2 任务

#### T17. 澄清文档结构定义

内容：

- 已确认项
- 未确认项
- 假设
- blocker
- 澄清问题

验收标准：

- 澄清文档结构清楚、可读

#### T18. 主 Agent 澄清提问与回答沉淀

内容：

- 主 Agent 可以在聊天中提出关键澄清问题
- 用户回答后可显式更新澄清文档

验收标准：

- 澄清问题和答案能进入正式产物

#### T19. 澄清文档编辑与版本

内容：

- 支持编辑
- 支持版本 / diff / 回滚

验收标准：

- 澄清文档不是一轮问答后就冻结

#### T20. 进入产品模型的推进门

内容：

- blocker 可见
- 当关键问题已足够明确时允许推进

验收标准：

- 用户能看懂当前为什么可推进或不可推进

优先级：

- `P1`

依赖：

- `T14`
- `T15`

## 8. 任务组六：产品模型阶段

### 8.1 目标

把需求定义转成产品骨架。

### 8.2 任务

#### T21. 产品模型文档结构定义

内容：

- 功能模块
- 页面列表
- 页面职责
- 用户流程
- 信息架构

验收标准：

- 产品模型文档可独立阅读

#### T22. 从结构化 + 澄清生成产品模型

内容：

- 支持显式生成产品模型
- 允许用户继续修改

验收标准：

- 产物不只是摘要，而是产品骨架

#### T23. 产品模型文档编辑与版本

内容：

- 编辑
- 版本
- diff
- 回滚

验收标准：

- 产品模型文档可持续演进

#### T24. 进入 PRD 的推进门

内容：

- 产品骨架稳定后，允许进入 PRD

优先级：

- `P1`

依赖：

- `T17`
- `T18`

## 9. 任务组七：PRD 与原型阶段

### 9.1 目标

把产品模型收敛成正式文档，再进一步表达为页面和交互原型。

### 9.2 任务

#### T25. PRD 文档生成与编辑

内容：

- 基于产品模型、结构化、澄清结果生成正式 PRD
- 支持编辑、版本、diff、回滚

验收标准：

- PRD 是正式文档，不是摘要

#### T26. 原型稿生成方案

内容：

- 定义原型交付形式
- 支持页面列表
- 支持页面结构
- 支持流转关系
- 支持交互说明

验收标准：

- 原型稿可用于理解页面与流程

#### T27. 原型稿查看与版本

内容：

- 支持查看原型结果
- 支持重新生成
- 支持版本化

验收标准：

- 原型阶段不是黑盒结果页

优先级：

- `P2`

依赖：

- `T22`
- `T23`

## 11. 推荐实现顺序

### 第一批：必须先打通

1. `T1-T7`
2. `T8-T12`
3. `T13-T16`

这一批完成后，产品就有一条真正可用的前半段主链：

- 正常聊天
- 需求整理文档
- 需求结构化文档

### 第二批：把理解层做完整

1. `T17-T20`
2. `T21-T24`

这一批完成后，产品定义的“理解层 + 定义层”会稳定下来。

### 第三批：补正式交付物

1. `T25-T27`
2. 导出能力

这一批完成后，才算真正进入“正式产品交付”能力。

## 12. 当前建议

如果只选最该立刻开做的一组，我建议是：

### 第一优先级

- `T4` 聊天输入分类
- `T5` 主 Agent 自然语言输出规范
- `T6` 聊天显式文档动作
- `T10` 需求整理文档生成
- `T11` 需求整理文档编辑与版本
- `T13` 需求结构化文档定义

原因很简单：

这几项直接决定 AIPM 看起来到底是“正常好用的 AI 产品专家”，还是“流程很重但不好用的系统”。
