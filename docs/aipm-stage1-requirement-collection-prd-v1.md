# AIPM 第一阶段产品需求文档 V1

## 1. 这份文档的定位

这份文档只定义 `AIPM` 的第一阶段，不再试图一次覆盖全流程。

当前阶段的目标非常明确：

- 把散落的需求点、评论、补充说明、上传文件全部收集起来
- 由 AI 整理成一份清晰的“需求点文档”
- 支持继续编辑
- 支持版本记录、查看历史、回滚

这一阶段做稳之后，再继续往后推 `需求文档 / PRD / 原型 / UI 稿`。

## 1.1 主 Agent 总原则

虽然这份文档当前只定义第一阶段，但有一条产品原则要先定死：

- 不只是第一阶段，后续所有阶段都必须经过 `主 Agent` 跟进、审视和回复

也就是说：

- 用户当前在哪个阶段，主 Agent 就要理解该阶段上下文
- 用户输入新内容后，主 Agent 要先判断这是补充、修改、追问还是推进
- 当前阶段产物更新后，主 Agent 要给出阶段内反馈
- 当前阶段达到推进条件后，主 Agent 要在聊天中提示是否进入下一阶段

所以主 Agent 不是“第一阶段临时存在”，而是整条流程始终存在的总协作者。

## 2. 当前阶段的一句话定义

`AIPM` 第一阶段是一个本地优先的需求采集与整理工作台，用来把零散诉求持续沉淀成一份清晰、可编辑、可版本化的需求点文档。`

## 3. 为什么先只做第一阶段

当前最真实的产品问题，不是后面的 PRD 或 UI 稿能不能生成，而是：

- 需求输入很散
- 来源很多
- 评论和需求混在一起
- 文档容易漏读
- 整理结果容易丢点
- 用户不知道当前内容是否已经足够

如果第一阶段没做好，后面的 `需求文档 / PRD / 原型 / UI` 都会建立在不可靠的输入上。

所以当前正确策略是：

- 先把第一阶段做扎实
- 先把“收集 + 整理 + 编辑 + 版本化”做好
- 再继续把后续阶段接上

## 4. 当前阶段解决的问题

第一阶段主要解决这几类问题：

1. 用户输入的需求可能非常散
2. 里面既有需求点，也有评论、担心、建议、背景说明
3. 还会有上传文件，例如 `md / txt / pdf / doc / xls`
4. 用户需要看到 AI 如何整理这些来源
5. 用户需要对整理后的结果继续修改
6. 用户需要查看历史版本，必要时回滚

## 5. 当前阶段的目标

### 5.1 用户目标

- 我输入一句话或几段零散内容后，系统能先把它们收住
- 我上传文件后，系统能保留文件并尽可能读取内容
- 我能看到原始来源，不会丢信息
- AI 能帮我把这些内容整理成一份清晰需求点文档
- 我可以手工继续修改这份文档
- 我可以查看历史版本和回滚

### 5.2 系统目标

- 为后续阶段提供稳定、可信的输入底稿
- 让第一阶段本身就成为一个可独立成立的产品能力

## 6. 第一阶段的输入

输入分三类：

### 6.1 用户直接输入

例如：

- “我想做一个任务管理应用，帮助团队协作和追踪项目进度”
- “第一版先做 Web 端，不做移动端”
- “要支持看板、任务指派、进度追踪”

### 6.2 沟通记录

包括：

- 用户补充说明
- 用户评论
- 用户对需求的质疑
- 用户对目标和范围的修正
- 某条记录的回复

这里的原则是：

- 沟通记录不等于正式需求
- 但必须完整保留，因为它们会影响整理结果

### 6.3 上传文件

支持：

- `txt`
- `md`
- `json`
- `csv`
- `pdf`
- `doc`
- `docx`
- `xls`
- `xlsx`

要求：

- 文件必须保留原件
- 系统尽可能提取正文
- 解析失败也要保留元数据和原文件

## 7. 第一阶段的核心产物

第一阶段只有一个核心正式产物：

- `需求点文档`

它的作用不是正式 PRD，而是：

- 把散落来源整理成一份清晰、连续、可编辑的需求点说明

### 7.1 需求点文档建议结构

建议包含：

1. 产品/需求概述
2. 当前已识别的目标
3. 当前已识别的核心场景
4. 当前已识别的功能诉求
5. 当前约束与边界
6. 风险与待确认项
7. 原始来源摘要

### 7.2 需求点文档的要求

- 可读
- 可编辑
- 可版本化
- 可继续 AI 整理
- 可作为下一阶段输入

## 8. 第一阶段的工作流

推荐工作流如下：

1. 用户输入需求点或上传文件
2. 系统保存原始来源
3. AI 读取全部来源
4. AI 生成来源摘要
5. AI 整理成需求点文档
6. 用户查看并继续编辑
7. 用户可继续补充来源
8. 用户主动触发重新整理
9. 文档版本被记录
10. 用户确认需求点已清晰后，再进入下一阶段

## 9. AI 在第一阶段到底做什么

AI 在第一阶段至少要做 4 件事：

### 9.1 来源理解

- 识别用户输入中的需求点
- 区分需求、评论、建议、背景说明
- 读取上传文档和来源记录

### 9.2 来源整理

- 生成 `source summary`
- 汇总多条记录和多份文件
- 避免遗漏后面的来源

### 9.3 需求点整理

- 把来源整理成一份清晰文档
- 抽取目标、场景、功能、约束、待确认项

### 9.4 协作指导

- 告诉用户当前缺什么
- 提示下一步建议补什么
- 当内容足够时，提示可以进入下一阶段

## 10. UI 交互要求

当前阶段的 UI 应以“需求采集工作区”为核心。

### 10.1 主界面

主界面保持三栏：

- 左侧：项目列表
- 中间：聊天协作区
- 右侧：阶段区

### 10.2 第一阶段的主工作方式

用户点击 `需求采集` 阶段后，打开全屏需求采集工作区。

全屏工作区分两部分：

- 左侧主区域：需求点文档
- 右侧来源区域：原始沟通记录和上传文件

### 10.3 左侧主区域

要求：

- 富文本编辑
- 编辑 / 预览切换
- AI 整理按钮
- 保存状态
- 版本历史
- 查看 diff
- 回滚版本

### 10.4 右侧来源区域

要求：

- 显示全部沟通记录
- 支持新增、编辑、删除、回复
- 显示全部上传文件
- 文件支持预览
- 记录修改后不自动重整文档
- 只有用户主动点击 `AI 整理` 才重新生成文档

### 10.5 聊天区

聊天区职责不是替代文档，而是协作整理：

- 用户继续补充需求
- AI 告诉用户当前识别到了什么
- AI 提示当前还缺什么
- AI 在用户要求时帮忙重新整理

聊天区要求：

- 输入后立即流式反馈
- 显示当前正在做什么
- 显示模型实时输出
- 显示最终建议

## 11. 版本管理要求

需求点文档必须支持版本管理。

至少支持：

- 保存历史版本
- 查看版本列表
- 查看版本时间
- 查看 diff
- 回滚到任意历史版本

版本来源至少分为：

- `manual`
- `ai`
- `rollback`

## 12. 本地保存要求

第一阶段所有数据都要按项目空间本地落盘。

建议目录：

```text
workspace/projects/<projectId>/
  sources/
    source-records.json
    files/
    file-index.json
  requirement-collection/
    requirement-points.md
    requirement-points.html
    source-summaries.json
    versions/
```

## 13. 进入下一阶段的条件

第一阶段不是“生成完就自动往下走”。

必须满足：

1. 已有一份清晰的需求点文档
2. 用户认为主要诉求已经被整合进去
3. 当前缺口已经明确列出
4. 主 Agent 判断“可以进入下一阶段”
5. 用户确认推进

## 14. 当前阶段不追求什么

为了避免目标发散，第一阶段先不追求：

- 直接生成 PRD
- 直接生成原型
- 直接生成 UI 稿
- 一次性走完整条链

这些是后续阶段的目标，不是第一阶段当前的主目标。

## 15. 当前产品判断

就目前来说，`AIPM` 最应该先做成的，不是“一个自动生成全套产物的系统”，而是：

`一个能把散落诉求和文件资料整理成清晰需求点文档，并支持持续修改和版本管理的 AI 需求采集工作台。`

这件事做好了，后面的阶段才会真正站得住。
