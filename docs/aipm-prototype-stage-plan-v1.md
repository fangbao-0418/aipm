# AIPM 原型阶段产品方案 v1

这份文档只定义 `原型` 这一阶段。

目标是把已经稳定的 `产品模型` 和 `PRD`，转成页面级、流程级、可查看可修改的交互表达。

---

## 1. 阶段定位

`原型` 是 AIPM 在 `PRD` 之后、`UI` 之前的交互表达阶段。

这一阶段不再主要回答：

- 需求是什么
- 产品结构是什么

而是开始回答：

- 页面怎么组织
- 用户怎么走流程
- 每个页面要承载什么
- 哪些地方需要标注交互和规则

一句话定义：

`把产品定义转成可查看、可讨论、可继续修改的交互原型。`

---

## 2. 这一阶段要解决的问题

即使已经有了：

- 产品模型
- PRD

后面仍然会出现这些问题：

- 页面到底长什么结构还不清楚
- 同一个功能，不同人脑中的页面形态不一样
- 文档写清了，但流程一落到页面上就断裂
- UI 还没开始做，但交互方案已经需要先被对齐

所以 `原型` 阶段解决的是：

1. 把 PRD 和产品模型转成页面表达
2. 把流程落到具体页面和区块
3. 让后续 UI 有稳定骨架

---

## 3. 阶段目标

### 3.1 用户目标

- 我希望看到这个产品到底会有哪些页面
- 我希望知道每个页面的结构和职责
- 我希望在做 UI 前就把流程和交互先看清楚

### 3.2 系统目标

- 为 UI 提供稳定页面骨架
- 为评审提供交互表达物
- 为后续标注和设计说明提供载体

---

## 4. 这一阶段的输入

主输入包括：

- `产品模型文档`
- `PRD 文档`

可参考：

- `需求结构化文档`
- `需求澄清文档`

原则上，原型应以前两者为主，不应脱离产品模型和 PRD 自由发挥。

---

## 5. 这一阶段的正式产物

这一阶段建议至少有两个正式产物：

- `原型文档 / 原型说明`
- `原型画布 / 页面稿`

如果第一版先收敛，也可以把它们先视为一个组合产物：

- `原型稿`

---

## 6. 原型阶段应该输出什么

第一版建议至少包含 5 类结果。

### 6.1 页面清单

回答：

- 当前产品有哪些核心页面或视图

### 6.2 页面结构

回答：

- 每个页面有哪些主要区块
- 各区块分别承担什么作用

### 6.3 页面间流转

回答：

- 用户会如何从一个页面进入另一个页面
- 主流程和关键分支是什么

### 6.4 关键交互说明

回答：

- 哪些交互是关键动作
- 哪些状态会影响用户判断

### 6.5 原型标注准备信息

回答：

- 哪些位置需要后续标注
- 哪些规则不能只靠画面表达

---

## 7. 原型原则

### 7.1 原型不是最终视觉稿

原型阶段的重点是：

- 结构
- 流程
- 交互

而不是颜色、品牌、细腻视觉。

### 7.2 页面要能承接 PRD

原型不能脱离 PRD 自由发挥。

页面、功能、流程必须能互相映射。

### 7.3 原型要可继续修改

原型不是一次产物，而是后续：

- 标注
- UI

的基础。

### 7.4 原型要方便沟通

这一阶段的原型不只是“看图”，还要方便：

- 评审
- 提问
- 修改

---

## 8. 主 Agent 在这一阶段怎么工作

### 8.1 主 Agent 的职责

主 Agent 在 `原型` 阶段主要做 4 件事：

1. 解释页面结构和流程
2. 回答交互相关问题
3. 根据用户反馈调整原型
4. 判断是否可以进入 UI

### 8.2 聊天行为

例如用户问：

- “这个页面是不是该拆成两个”
- “这条流程为什么从这里开始”
- “首页应该先展示什么”

主 Agent 应直接回答。

如果用户说：

- “生成原型”
- “调整这个页面结构”
- “把流程改顺一点”

主 Agent 才进入原型更新动作。

### 8.3 不默认自动重画原型

普通讨论不应自动改写正式原型产物。

只有用户显式要求时，才更新正式原型。

---

## 9. 什么时候生成原型

### 9.1 允许触发

- 用户确认从 PRD 进入原型阶段
- 用户点击“生成原型”
- 用户在聊天里要求“生成 / 更新原型”

### 9.2 不应自动触发

- 普通问答
- 单条小修改讨论
- 只是在浏览 PRD 时

---

## 10. 什么时候更新原型

原型更新应当是显式动作。

### 10.1 当上游变化后

如果：

- 产品模型变化
- PRD 变化

则原型应被标记为：

- 待更新

但不自动重画。

### 10.2 当用户要求更新后

只有用户明确要求，才重新生成正式原型。

---

## 11. UI 交互方案

### 11.1 主工作区

建议延续前面阶段的交互模式：

- 左侧主区：原型画布 / 页面稿
- 右侧辅助区：页面列表、流程、标注准备信息、版本

### 11.2 左侧主区

要求：

- 页面级原型展示
- 页面切换
- 画布可查看
- 支持修改入口

### 11.3 右侧辅助区

建议承载：

- 页面清单
- 页面职责
- 关键流程
- 后续标注提示
- 版本记录

### 11.4 聊天区

聊天区负责：

- 解释原型结构
- 回答交互问题
- 接收原型修改指令
- 提示是否可以进入 UI

---

## 12. 交付形式建议

第一版建议原型采用：

- HTML 页面稿
- 页面低保真结构
- 可点击切换页面
- 右侧配合文字说明

不建议第一版就追求：

- 复杂白板编辑器
- 完整 Figma 替代品

---

## 13. 与下一阶段的关系

`原型` 的主要去向是：

- `UI`

进入 UI 的前提是：

- 页面和流程已经稳定
- 用户认可当前原型骨架

---

## 14. 完成标准

当满足以下条件时，可认为 `原型` 阶段完成：

1. 已形成正式原型稿
2. 页面清单、页面结构、主要流程已清楚
3. 原型与 PRD 和产品模型保持一致
4. 用户认可可以进入 UI

---

## 15. 第一版范围建议

第一版优先做：

- 页面清单
- 页面结构稿
- 页面流转
- 可查看的 HTML 原型
- 可重新生成
- 版本记录

第一版先不追求：

- 完整图形编辑器
- 高自由度拖拽编辑
- 复杂动画和状态系统

---

## 16. 结论

`原型` 阶段的意义，是把 PRD 和产品模型变成真正能被看懂、讨论、修改的页面和交互表达。

这一步一旦稳定，后续 UI 就不会从零开始猜。
