# AIPM 四阶段流转方案 v1

这份文档定义前四个核心阶段的完整流转：

1. 需求整理
2. 需求结构化
3. 需求澄清
4. 产品模型

目标是把 AIPM 第一条真正稳定的产品主链讲清楚。

---

## 1. 为什么先只看前四阶段

后面还有：

- PRD
- 原型
- UI

但前四阶段是整个产品定义的基础。

如果前四阶段不清楚，后面的正式文档和设计表达都会不稳定。

---

## 2. 四阶段一句话定义

### 2.1 需求整理

把散乱来源整理成一份可读文档。

### 2.2 需求结构化

把可读文档转成结构化输入。

### 2.3 需求澄清

把关键缺口和模糊点显式化并收敛。

### 2.4 产品模型

把清楚的需求转成模块、页面、流程组成的产品骨架。

---

## 3. 四阶段上下游关系

### 3.1 需求整理 -> 需求结构化

`需求整理` 提供：

- 一份正式的需求整理文档

`需求结构化` 消耗它，产出：

- 目标
- 用户
- 场景
- 功能
- 边界
- 风险

### 3.2 需求结构化 -> 需求澄清

`需求结构化` 提供：

- 已明确项
- 未明确项

`需求澄清` 继续把关键缺口转成问题和确认结果。

### 3.3 需求澄清 -> 产品模型

`需求澄清` 提供：

- 已确认项
- 未确认项
- 假设
- blocker 状态

`产品模型` 基于这些较稳定的信息生成产品骨架。

---

## 4. 四阶段输入输出总表

### 4.1 需求整理

输入：

- 聊天补充
- 来源记录
- 上传文件

输出：

- 需求整理文档

### 4.2 需求结构化

输入：

- 需求整理文档

输出：

- 需求结构化文档

### 4.3 需求澄清

输入：

- 需求结构化文档
- 用户回答

输出：

- 需求澄清文档

### 4.4 产品模型

输入：

- 需求结构化文档
- 需求澄清文档

输出：

- 产品模型文档

---

## 5. 主 Agent 的统一规则

前四阶段都遵循相同的主 Agent 原则：

### 5.1 先回答，再决定是否进入文档动作

### 5.2 文档更新必须显式触发

### 5.3 阶段推进必须显式确认

### 5.4 来源、阶段产物、聊天消息三者分离

---

## 6. 四阶段推进条件

### 6.1 需求整理 -> 需求结构化

条件：

- 已有正式需求整理文档
- 当前文档与来源一致，或用户接受当前版本
- 用户确认进入下一阶段

### 6.2 需求结构化 -> 需求澄清

条件：

- 已有正式结构化文档
- 系统识别到仍有关键缺口，或用户主动进入澄清
- 用户确认进入

### 6.3 需求澄清 -> 产品模型

条件：

- blocker 已收敛
- 核心问题已有答案或可带假设继续
- 用户确认进入产品模型

### 6.4 产品模型 -> PRD

条件：

- 页面、模块、流程已经清楚
- 用户确认当前骨架可进入 PRD

---

## 7. 四阶段状态流转

每个阶段建议统一采用：

- 未生成
- 已生成
- 待更新
- 已确认

### 7.1 未生成

还没有正式产物。

### 7.2 已生成

已经有正式文档，但仍可继续修改。

### 7.3 待更新

来源或上游结果变化了，当前产物需要重新生成或整理。

### 7.4 已确认

用户认可当前产物可作为下一阶段输入。

---

## 8. 用户动作流

### 8.1 需求整理阶段

用户会：

- 输入需求
- 上传文件
- 补充记录
- 点击 AI 整理
- 编辑文档
- 看版本
- 回滚
- 确认进入结构化

### 8.2 需求结构化阶段

用户会：

- 查看结构结果
- 询问结构问题
- 重新结构化
- 编辑结构文档
- 看版本
- 确认进入澄清

### 8.3 需求澄清阶段

用户会：

- 查看澄清问题
- 回答问题
- 调整假设
- 更新澄清文档
- 看 blocker
- 确认进入产品模型

### 8.4 产品模型阶段

用户会：

- 查看模块、页面、流程
- 询问结构合理性
- 调整模型
- 更新模型文档
- 确认进入 PRD

---

## 9. 系统动作流

### 9.1 需求整理

系统负责：

- 保存来源
- 标记文档是否待整理
- 在显式触发后生成正式文档

### 9.2 需求结构化

系统负责：

- 基于整理文档提取结构信息
- 生成结构化文档
- 标记待确认项

### 9.3 需求澄清

系统负责：

- 识别关键缺口
- 形成问题列表
- 更新已确认 / 未确认 / 假设 / blocker

### 9.4 产品模型

系统负责：

- 生成模块、页面、流程骨架
- 形成统一模型文档
- 标记模型假设

---

## 10. 四阶段核心交付物

### 10.1 第一阶段

- 需求整理文档

### 10.2 第二阶段

- 需求结构化文档

### 10.3 第三阶段

- 需求澄清文档

### 10.4 第四阶段

- 产品模型文档

这四份文档共同构成后续 `PRD / 原型 / UI` 的输入基础。

---

## 11. 第一版实现建议

第一版应围绕这四阶段做最小可用闭环：

1. 聊天正常回答
2. 第一阶段可整理正式文档
3. 第二阶段可生成结构化结果
4. 第三阶段可形成澄清结果
5. 第四阶段可生成产品骨架
6. 每阶段可编辑、版本化、显式推进

---

## 12. 结论

`需求整理 -> 需求结构化 -> 需求澄清 -> 产品模型` 应该是 AIPM 第一条完整稳定的产品定义主链。

只要这四阶段跑顺，后面的：

- PRD
- 原型
- UI

才会真正有可靠基础。
